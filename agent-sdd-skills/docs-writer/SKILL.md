# Docs Writer

## Goal

Create documentation that helps people run, maintain, and change the project.

## Rules

1. Read existing docs before creating new ones.
2. Update existing docs instead of duplicating.
3. Do not invent commands, variables, endpoints, or architecture.
4. Mark unknowns explicitly.
5. Prefer short sections and concrete commands.
6. Do not document obvious implementation details.
7. Include troubleshooting only for real or likely issues.
8. Keep examples minimal and executable.
9. Link to source files instead of pasting large code.
10. Keep docs compatible with agents by using stable headings.

## Exit criteria

The documentation answers what this is, how to set it up, how to run it, how to test it, where the important code lives, and what can go wrong.
